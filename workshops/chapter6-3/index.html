    Performance Measurement and Improvement: Building Learning Systems

    So you've built specialized retrievers and a router to connect them. Congratulations! Now comes the part that separates good systems from great ones: measurement and improvement.

    This chapter is about turning your RAG system into something that gets better every week, not something that slowly rots in production.

    Prerequisites and context

    • Read: 6.1 Query Routing Foundations, 6.2 Tool Interfaces and Implementation
    • You have specialized tools and a router in place

    Learning objectives

    • Define routing and retrieval metrics that matter
    • Build dashboards that localize bottlenecks
    • Turn user interactions into training data to improve routing and tools

    Testing Query Routing: The Metrics That Actually Matter

    Let me tell you about the most expensive mistake I ever made. We spent three months improving our document search to near perfection - 95% accuracy! Users still complained constantly.

    Why? Our router only sent document queries to document search 40% of the time. We'd optimized the wrong thing.

    The Metrics You Need (Not the Ones You Want)

    Here's what you actually need to measure:

    Tool Precision: When you select a tool, is it the right one? Tool Recall: When a tool should be selected, do you select it? Tool F1: The harmonic mean (because both matter) Per-Tool Recall: The killer metric everyone forgets

    That last one saved my bacon. Here's real data from a construction system:

    Tool Times Needed Times Selected Per-Tool Recall
    Blueprint Search 523 498 95.2%
    Document Search 412 387 93.9%
    Schedule Search 234 198 84.6%
    Permit Search 67 12 17.9%
    Compliance Check 23 2 8.7%

    See the problem? The rarely-used tools had terrible recall. Users searching for permits got frustrated. We thought the system was "mostly working" because the common tools worked well.

    The Data Leakage Trap That Everyone Falls Into

    Here's a mistake that's embarrassing in retrospect. We had this few-shot example in our router:

    "Find the electrical permit for the Johnson building" → PermitSearch

    Then we had this in our test set:

    "Find the electrical permit for the Johnson building" → PermitSearch

    Guess what? 100% accuracy on that test case! In production? Users asked "Where's the Johnson building electrical permit?" and got blueprint search. Different wording, total failure.

    The fix: Keep your test queries completely separate from your few-shot examples. Better yet, use real user queries for testing (after you have them).

    The Simple Code That Reveals Everything

    Here's the evaluation code I use on every project:

    def evaluate_router(test_cases, router):
    tool_stats = defaultdict(lambda: {"needed": 0, "selected": 0})

    for case in test_cases:
        expected = set(case["expected_tools"])
        predicted = set(router.route(case["query"]))

        # Track per-tool stats
        for tool in expected:
            tool_stats[tool]["needed"] += 1
            if tool in predicted:
                tool_stats[tool]["selected"] += 1

    # Calculate per-tool recall
    for tool, stats in tool_stats.items():
        recall = stats["selected"] / stats["needed"]
        print(f"{tool}: {recall:.1%} ({stats['selected']}/{stats['needed']})")

    Run this weekly. The tools with low recall are where users are getting frustrated.

    User Interfaces: The Secret Weapon Nobody Talks About

    Here's the insight that changed how I build RAG systems: tools good enough for language models are usually good enough for humans too.

    The Google Lesson We All Missed

    Think about how search works in these other applications. Each tool has different UI and different filters to allow you to find the right tools. You can think of YouTube as Google's video index and Google Maps as the directions index. LinkedIn is your network and professional index, and then Google for everything else.

    But even then, whether or not Google thinks your search request is gonna be a video or a map or directions, it's gonna put you in the right place. Google is very opinionated with what kind of UI they wanna show you based on the search request you give it.

    We can do the same thing. That construction company? We built:

    • A chat interface for general questions
    • A blueprint search page with date filters
    • A document search with type filters
    • A schedule viewer with Gantt charts

    Here's the key insight: if we know our user is an expert, we can shift the probability that we select the right tool to a hundred percent if we think they know what they want. And if you can, why delegate to a model to begin with?

    Usage split: 60% direct tools, 40% chat. The experts went straight to specialized tools. New users started with chat, then graduated to direct tools. Just like I don't force myself to Google everything - if I know I want directions, I just go to Maps. If I know I want videos, I just go to YouTube.

    The Dual-Mode Interface Pattern

    Here's the pattern that works:

    ┌─────────────────────────────────────┐
│          Unified Search Bar          │
│  "What are you looking for today?"   │
└─────────────────────────────────────┘
                  │
     ┌────────────┴────────────┐
     ▼                         ▼
┌─────────────┐         ┌─────────────┐
│    Chat     │         │   Toolbox   │
│  Interface  │         │             │
│             │         │ 📋 Documents │
│ "Ask me     │         │ 📐 Blueprints│
│  anything"  │         │ 📅 Schedules │
│             │         │ 📜 Permits   │
└─────────────┘         └─────────────┘

    Users self-select based on their needs and expertise. Both interfaces use the same underlying tools.

    The Unexpected Benefits

    What surprised me:

    1. Experts are faster with direct tools - 3x faster than chat for experienced users
    2. Chat teaches tool discovery - Users learn what's possible, then switch to direct access
    3. Clear capabilities reduce frustration - "Oh, there's a permit search? Let me try that!"
    4. Direct tools provide better training data - You know exactly what users want

    That last point is huge. When someone uses blueprint search directly, that's ground truth for router training. More interestingly, we can use the human interactions on this document selection, right? If we search for a blueprint and they click a blueprint, that becomes training that we can use to improve our ranking models or our search models.

    Here's the real insight: Building a RAG application, building a chat bot is a feature, not a benefit. Most people want to use a tool like this to be able to answer questions and figure out where the data is. Building search is gonna be very important for them.

    User Feedback as Training Data: The Goldmine in Your Logs

    Every user interaction is a potential training example. Most teams throw this gold away.

    The Feedback Collection That Actually Works

    Forget complex feedback forms. Here's what actually gets used:

    Click tracking: Which result did they click? That's relevance feedback. Tool selection: Which specialized tool did they choose? That's routing feedback. Query refinement: How did they change their query? That's intent clarification. Time to success: How long until they found what they needed? That's overall quality.

    We built this simple tracking:

    @app.post("/feedback")
def track_feedback(
    query: str,
    selected_tool: str,
    clicked_results: List[str],
    time_to_click: float
):
    # Log everything
    feedback_log.append({
        "timestamp": datetime.now(),
        "query": query,
        "tool": selected_tool,
        "clicks": clicked_results,
        "speed": time_to_click
    })

    # Fast interactions = success
    if time_to_click < 10.0 and len(clicked_results) > 0:
        # Add to training examples
        training_queue.add({
            "query": query,
            "correct_tool": selected_tool
        })

    After 30 days: 10,000 new training examples, zero manual labeling.

    The Virtuous Cycle in Action

    Here's what happened at that construction company:

    Week 1: 65% routing accuracy, lots of user frustration Week 4: Added top 100 failure cases as examples, 78% accuracy Week 8: Dynamic example selection from successful queries, 85% accuracy Week 12: Tool-specific improvements based on usage patterns, 91% accuracy Week 16: Users barely use chat anymore, direct tools handle 75% of queries

    The system literally trained itself using user behavior.

    Integration points

    • Router evaluation connects directly to per-tool retrieval metrics
    • Feedback instrumentation ties to 3.1 Feedback Collection and streaming UX in 3.2
    • Improvements inform production priorities in 7

    The Success Formula Redux

    Remember our formula from earlier chapters? With routing, it becomes even more powerful:

    P(success) = P(find right doc | right tool) × P(right tool)

    This equation can really help you identify the limiting factors of your system. You generally just want to have a dashboard that prints out both the conditional probability of finding that chunk and the probability of high recall.

    Once you know, you'll know what to work on. Do I want to focus on the subsystem or do I want to focus on the router? If you want to focus on the router, you have to think about things like few shotting and descriptions. Whereas if you wanna improve the index, you might wanna think about things like fine tuning, generally more training data and improving the way you do filtering.

    Now we can build dashboards that show exactly where to focus:

    Blueprint Search: 92% retrieval × 95% routing = 87% success ✓
Document Search:  88% retrieval × 94% routing = 83% success ✓
Schedule Search:  95% retrieval × 85% routing = 81% success ⚠️
Permit Search:    78% retrieval × 18% routing = 14% success ❌

    One glance tells you: fix permit routing first, then permit retrieval. No guesswork.

    The Weekly Standup That Actually Helps

    Every Monday, my teams answer three questions:

    1. Which tool has the lowest routing recall? (Fix examples)
    2. Which correctly-routed tool has the lowest retrieval? (Fix the tool)
    3. Which successful patterns can we replicate? (Scale what works)

    This turns vague "make it better" meetings into concrete action items.

    Building Improvement Flywheels

    The best RAG systems improve themselves. Here's how to build that flywheel:

    Step 1: Instrument Everything

    def track_interaction(func):
    def wrapper(query, *args, **kwargs):
        start = time.time()
        result = func(query, *args, **kwargs)
        duration = time.time() - start

        log_event({
            "function": func.__name__,
            "query": query,
            "duration": duration,
            "success": result.success,
            "result_count": len(result.data)
        })

        return result
    return wrapper

    Apply this to every tool and router call. Data is ammunition for improvement.

    Step 2: Automate the Obvious

    Every week, automatically:

    • Add successful queries as few-shot examples
    • Flag tools with declining performance
    • Generate reports on new query patterns
    • Test against regression suite

    A client saved 20 hours/week just by automating their "what worked this week" analysis.

    Step 3: Close the Loop

    User clicks result → Mark as relevant → Add to training data → Improve retrieval → User finds results faster → More clicks → Better training data

    This is the flywheel. Once it starts spinning, it accelerates itself.

    The Mistakes Everyone Makes (So You Don't Have To)

    Measuring aggregate metrics only: "System accuracy is 75%" tells you nothing. Decompose everything.

    Ignoring rare tools: That compliance checker used twice a month? When someone needs it, they REALLY need it.

    Optimizing for robots, not humans: 95% technical accuracy means nothing if users can't find what they need.

    Waiting for perfect data: Start collecting feedback on day 1. Messy data beats no data.

    Over-engineering the feedback loop: Thumbs up/down beats complex ratings. Time-to-click beats satisfaction surveys.

    What Success Actually Looks Like

    At that construction company, after 6 months:

    • Support tickets about "can't find" dropped 78%
    • Expert users stopped using chat entirely (good!)
    • New tools could be added in hours, not weeks
    • The system improved without engineering effort

    But the best metric? The foreman told me: "I stopped calling the office. I just find what I need."

    That's success.

    The Path Forward

    This generally concludes the course. Obviously there's gonna be many more office hours, and I'll still be on Slack for the remainder of the year to answer any questions. But what I hope to distill in you is that you need evaluations.

    Way too many teams I work with have either no evaluations or a tiny set of evaluations, like 10 or 20 examples. But evaluations are critical to understanding how to improve your system. Evals represent the dataset you can use to inform your decision making.

    Ideally you can change the way you run meetings so that your conversations are not just about how to make the AI better, but how to move specific metrics.

    You now have all the pieces:

    • Specialized retrievers for different content types
    • Clear tool interfaces
    • Smart routing with good examples
    • Measurement that shows what to fix
    • User feedback that trains itself

    The secret isn't in any single component. It's in connecting them so the system gets better every week.

    The Fundamental Truth About Machine Learning

    One of the biggest lessons I hope you can take away is the value of synthetic data. Synthetic data and customer feedback is ultimately what you need to make your applications go to the next level. This is the fundamental building block of creating good and successful machine learning products.

    And if you refuse to believe this, you're ultimately condemning yourself to being lost and confused in this very hyped up space of machine learning. And it's been the same every single time. There are always going to be new companies, new technologies, and new frameworks with new names. But we're all more or less doing the same thing we have been doing for the past 20 years.

    Ultimately the process has been the same: - A good product generates better evaluations with strong user experience, good UI and good expectation setting - Better evaluations allow you to train and fine tune models to create a better product - Data analysis over your users (especially segmentation) tells you where to focus your product development efforts

    And that process is ultimately what building and deploying a machine learning based project is all about.

    Remember: The goal isn't perfection. It's building something that improves faster than user needs grow. Nail that, and you've won.
